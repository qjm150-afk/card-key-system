#### 2.5.1 设计背景与定位

**核心定位**：**卡种 = 分组统计容器 + 预览配置**

```
┌─────────────────────────────────────────────────────────────────┐
│                        卡种管理体系                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   卡种（分组容器）                                               │
│   ├── 基础信息：名称、描述                                       │
│   ├── 预览配置：预览图、模糊程度（可选）                          │
│   └── 统计数据：库存、已用、销售数据                              │
│                                                                 │
│   卡券（独立属性）                                               │
│   ├── 卡密信息：卡号、密码、激活状态                              │
│   ├── 过期设置：过期时间/天数（独立管理）                         │
│   ├── 内容设置：飞书链接、设备限制（独立设置）                    │
│   └── 销售信息：销售状态、订单号、渠道                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**功能边界**：

| 功能 | 是否属于卡种 | 说明 |
|-----|-------------|------|
| 分组统计 | ✅ 是 | 按卡种统计库存、已用数、销售数据 |
| 预览配置 | ✅ 是 | 不同卡种可配置不同预览图 |
| 过期设置 | ❌ 否 | 每张卡券独立管理过期时间 |
| 飞书链接 | ❌ 否 | 每张卡券独立设置链接 |
| 设备限制 | ❌ 否 | 每张卡券独立设置 |
| 卡种分类 | ❌ 否 | 通过创建不同卡种实现分类（如"对外销售"、"内部测试"） |

**与现有实现的差异**：

| 项目 | 现有实现（错误） | 正确设计 |
|-----|----------------|---------|
| 卡种分类 | 有分类下拉框 | 移除，通过创建不同卡种实现 |
| 过期设置 | 卡种层面设置 | 卡券独立管理 |
| 内容设置 | 卡种层面设置 | 卡券独立设置 |
| 属性继承 | 卡券继承卡种属性 | 卡券完全独立，仅通过 card_type_id 关联 |

#### 2.5.2 核心优势

| 优势 | 说明 |
|-----|------|
| **统计清晰** | 按卡种汇总库存、已用数、销售数据 |
| **预览差异化** | 不同卡种可配置不同预览图，提高转化 |
| **灵活分类** | 通过命名区分用途，无需繁琐分类选项 |
| **卡券独立** | 每张卡券属性完全独立，无继承复杂性 |

#### 2.5.3 数据库设计

**卡种表 (card_types)** - 简化版：

```sql
CREATE TABLE card_types (
    id SERIAL PRIMARY KEY,
    
    -- 基础信息
    name VARCHAR(200) NOT NULL COMMENT '卡种名称',
    description TEXT COMMENT '描述说明',
    
    -- 预览设置（核心功能）
    preview_enabled BOOLEAN DEFAULT false COMMENT '是否启用预览',
    preview_image TEXT COMMENT '预览截图URL',
    blur_level INTEGER DEFAULT 8 COMMENT '模糊程度(px): 4/8/12',
    
    -- 统计（冗余字段，提升查询性能）
    total_count INTEGER DEFAULT 0 COMMENT '总卡券数',
    used_count INTEGER DEFAULT 0 COMMENT '已使用数',
    unsold_count INTEGER DEFAULT 0 COMMENT '未售出数',
    sold_count INTEGER DEFAULT 0 COMMENT '已售出数',
    refunded_count INTEGER DEFAULT 0 COMMENT '已退款数',
    disputed_count INTEGER DEFAULT 0 COMMENT '有纠纷数',
    
    -- 状态
    status INTEGER DEFAULT 1 COMMENT '状态：1=有效, 0=无效',
    
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);
```

**卡券表 (card_keys_table)** - 保持现有字段不变：

```sql
-- 卡券表保持原有字段，新增 card_type_id 关联卡种
ALTER TABLE card_keys_table ADD COLUMN card_type_id INTEGER REFERENCES card_types(id);

-- 卡券原有字段（完全独立，不继承）：
-- key_value, status, feishu_url, feishu_password, link_name
-- expire_at, expire_after_days, activated_at, max_devices, devices
-- sale_status, order_id, sales_channel, user_note...
```

#### 2.5.6 批量操作逻辑设计

**激活状态说明**：

| 状态 | 判断条件 | 数据来源 |
|-----|---------|---------|
| **已停用** | `status = 0` | 数据库存储 |
| **有效** | `status = 1` 且未使用过（devices为空且used_count=0） | 前端计算 |
| **已激活** | `status = 1` 且已使用过（devices不为空或used_count>0） | 前端计算 |

**卡券创建默认状态**：

所有卡券（无论过期类型）创建时统一 `status = 1`（有效），前端根据使用情况自动显示状态。

| 过期类型 | 创建时 status | 创建时显示 | 首次验证后 |
|---------|--------------|-----------|-----------|
| 固定日期 | `1` | 有效 | 已激活 |
| 永久有效 | `1` | 有效 | 已激活 |
| 按激活天数 | `1` | 有效 | 已激活 + 记录 `activated_at` |

**批量激活逻辑**：

批量激活仅用于：将手动停用（`status=0`）的卡券重新启用（`status=1`）。

| 过期类型 | 批量激活 | 批量停用 | 说明 |
|---------|---------|---------|------|
| 固定日期 | ✅ 可以 | ✅ 可以 | 设置 `status = 1` |
| 永久有效 | ✅ 可以 | ✅ 可以 | 设置 `status = 1` |
| 按激活天数 | ✅ 可以 | ✅ 可以 | 设置 `status = 1`，但**不重置** `activated_at` |

**交互方案**：

点击「批量激活」时，如果选中的卡券中有"按激活天数过期"且已被使用过的卡券，弹窗提示：

```
┌─────────────────────────────────────────────────────────────────────┐
│ 提示                                                           [×] │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│ 选中的卡券中有 N 张"按激活天数过期"的卡券：                         │
│                                                                     │
│ • 已使用过的卡券：重新启用后，过期时间不变（不重置激活时间）          │
│ • 未使用过的卡券：重新启用后，首次验证时开始计算过期时间             │
│                                                                     │
│ 是否继续操作？                                                      │
│                                                                     │
├─────────────────────────────────────────────────────────────────────┤
│                                          [取消]  [确认激活]         │
└─────────────────────────────────────────────────────────────────────┘
```

**用户首次验证逻辑（按激活天数过期）**：

```
用户输入卡密验证
    ↓
检查 expire_type == 'relative'
    ↓
检查 activated_at 是否为空
    ↓
为空 → 
    1. 设置 activated_at = 当前时间
    2. 绑定设备
    3. 验证成功
    ↓
不为空 → 正常验证（检查是否过期）
```
