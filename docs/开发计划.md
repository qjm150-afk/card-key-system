#### 2.5.1 架构背景

**参考案例**：借鉴成熟发卡系统的 **卡种 → 卡券** 两级结构

**当前问题**：
- 卡密管理为扁平结构，所有卡密独立设置属性
- 无法批量设置同一批次卡密的统一属性（过期时间、飞书链接等）
- 视图隔离、预览配置等功能缺乏统一入口

**解决方案**：引入卡种概念，实现分层管理

```
┌─────────────────────────────────────────────────────────────────┐
│                        卡种管理体系                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   卡种（容器）                                                   │
│   ├── 统一属性：名称、过期设置、飞书链接、预览配置、设备限制        │
│   ├── 统计数据：库存、已用、总数                                  │
│   └── 卡券（具体卡密）                                          │
│       └── 个体属性：卡号、密码、激活状态、买家信息、激活时间        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 2.5.2 核心优势

| 优势 | 说明 |
|-----|------|
| **属性继承** | 卡券自动继承卡种的统一属性，无需逐个设置 |
| **视图天然隔离** | 不同卡种 = 不同视图，无需额外标签页 |
| **批量操作简化** | 修改卡种属性即可批量更新所有卡券 |
| **统计更直观** | 卡种层面汇总库存、已用数 |
| **灵活配置** | 不同卡种可配置不同过期方式、预览图等 |

#### 2.5.3 数据库设计

**新增：卡种表 (card_types)**

```sql
CREATE TABLE card_types (
    id SERIAL PRIMARY KEY,
    
    -- 基础信息
    name VARCHAR(200) NOT NULL COMMENT '卡种名称',
    type VARCHAR(20) DEFAULT 'unique' COMMENT '卡种类型：unique=唯一卡, multi=多次卡',
    category VARCHAR(50) COMMENT '卡种分类：external=对外销售, internal=内部使用',
    description TEXT COMMENT '描述说明',
    
    -- 过期设置（统一在卡种层面）
    expire_type VARCHAR(20) DEFAULT 'fixed' COMMENT '过期类型：fixed=固定日期, relative=按激活天数, permanent=永久',
    expire_days INTEGER COMMENT '过期天数（仅 relative 类型）',
    expire_at TIMESTAMP COMMENT '过期时间（仅 fixed 类型）',
    
    -- 内容设置
    feishu_url TEXT COMMENT '飞书链接',
    feishu_password VARCHAR(100) COMMENT '访问密码',
    link_name VARCHAR(100) COMMENT '链接名称',
    
    -- 设备限制
    max_devices INTEGER DEFAULT 5 COMMENT '最大设备数',
    
    -- 预览设置（用于预览转化模式）
    preview_image TEXT COMMENT '预览截图URL',
    blur_level INTEGER DEFAULT 8 COMMENT '模糊程度(px)',
    preview_enabled BOOLEAN DEFAULT false COMMENT '是否启用预览',
    
    -- 统计
    total_count INTEGER DEFAULT 0 COMMENT '总数',
    used_count INTEGER DEFAULT 0 COMMENT '已用数',
    
    -- 状态
    status INTEGER DEFAULT 1 COMMENT '状态：1=有效, 0=无效',
    sort_order INTEGER DEFAULT 100 COMMENT '排序',
    
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);
```

**字段继承逻辑**：

```python
def get_card_property(card_key, card_type, property_name):
    """
    获取卡券属性值（优先使用卡券自身值，否则继承卡种值）
    """
    card_value = getattr(card_key, property_name)
    if card_value is not None:
        return card_value
    return getattr(card_type, property_name)
```

#### 2.5.6 数据迁移策略

**现有数据迁移**：

```sql
-- 1. 创建默认卡种（包含所有现有卡密）
INSERT INTO card_types (name, category, expire_type, expire_at, feishu_url, feishu_password, link_name, max_devices, total_count, used_count)
SELECT 
    '历史卡密' as name,
    'external' as category,
    'fixed' as expire_type,
    MAX(expire_at) as expire_at,
    feishu_url,
    feishu_password,
    link_name,
    5 as max_devices,
    COUNT(*) as total_count,
    COUNT(CASE WHEN status = 0 THEN 1 END) as used_count
FROM card_keys_table
WHERE feishu_url IS NOT NULL
GROUP BY feishu_url, feishu_password, link_name;

-- 2. 关联卡券到卡种
UPDATE card_keys_table ck
SET card_type_id = ct.id
FROM card_types ct
WHERE ck.feishu_url = ct.feishu_url
  AND (ck.feishu_password = ct.feishu_password OR (ck.feishu_password IS NULL AND ct.feishu_password IS NULL));
```

- 现有 API 保持兼容，无 `card_type_id` 的卡密使用原有逻辑

**现有逻辑** (`src/main.py:validate_card_key()`):

**⚠️ 影响分析**：

| 影响点 | 改造需求 | 兼容策略 |
|-------|---------|---------|
| 需要JOIN查询卡种 | 改为两步查询（先查卡密，再查卡种） | 卡密无card_type_id时使用原逻辑 |
| 过期时间继承 | 优先使用卡券expire_at，为空则根据卡种计算 | expire_type=relative时动态计算 |
| 飞书链接继承 | 优先使用卡券feishu_url，为空则用卡种 | 保持现有字段作为覆盖值 |
| 设备限制继承 | 优先使用卡券max_devices，为空则用卡种 | 默认值5作为最终兜底 |

**改造后逻辑**：
```python
# 1. 查询卡密
card_data = client.table('card_keys_table').select('*').eq('key_value', card_key).execute()

# 2. 如果关联了卡种，获取卡种信息
if card_data.get('card_type_id'):
    card_type = client.table('card_types').select('*').eq('id', card_data['card_type_id']).execute()
    # 属性继承逻辑
    feishu_url = card_data.get('feishu_url') or card_type.get('feishu_url')
    max_devices = card_data.get('max_devices') or card_type.get('max_devices', 5)
    
    # 过期时间计算（新增按天数过期逻辑）
    if card_data.get('expire_at'):
        expire_at = card_data['expire_at']
    elif card_type.get('expire_type') == 'relative':
        # 按激活天数计算
        activated_at = card_data.get('activated_at') or card_data.get('last_used_at')
        if activated_at:
            expire_at = activated_at + timedelta(days=card_type['expire_days'])
    elif card_type.get('expire_type') == 'fixed':
        expire_at = card_type.get('expire_at')
else:
    # 兼容模式：无卡种的卡密使用原逻辑
    feishu_url = card_data.get('feishu_url')
    max_devices = card_data.get('max_devices', 5)
    expire_at = card_data.get('expire_at')
```

### 4.1 功能概述

#### 4.2.2 泄露检测逻辑

```
泄露风险评分 = f(外部访问比例, 异常访问时段, 地域分散度, 分享次数)

风险等级：
- 低风险 (0-30分)：外部访问比例 < 10%，无异常行为
- 中风险 (31-70分)：外部访问比例 10-30%，或存在少量异常
- 高风险 (71-100分)：外部访问比例 > 30%，或明显异常行为

预警触发条件：
1. 外部访问比例突增（较前日增长 > 50%）
2. 单日外部访问人数超过卡密有效数量
3. 非预期时段（如深夜）大量访问
4. 新地域突然出现大量访问
```

### 4.5 风险与应对

| 风险 | 影响 | 应对措施 |
|-----|------|---------|
| 飞书个人版不支持统计API | 功能无法实现 | 考虑升级企业版，或使用替代方案（如自建访问统计） |
| API调用频率限制 | 数据同步延迟 | 增量同步、批量查询、本地缓存 |
| 统计数据延迟 | 预警不及时 | 设置合理预期，增加其他检测维度 |
| Token过期处理 | 服务中断 | 自动刷新机制 + 提前预警 |

### 4.6 替代方案

如果飞书API无法满足需求，考虑以下替代方案：

1. **自建访问统计**
   - 在iframe加载时记录访问日志
   - 通过Referer判断内嵌/外部访问
   - 局限：只能统计本系统内的访问，无法检测直接访问飞书链接的情况

2. **第三方统计工具**
   - 集成百度统计/Google Analytics
   - 局限：同样只能统计本系统内访问

3. **飞书企业版升级**
   - 获取完整API能力
   - 局限：增加成本
